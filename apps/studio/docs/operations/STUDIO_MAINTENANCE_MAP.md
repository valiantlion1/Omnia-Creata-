## Recent stabilization wins

- `.105` closes another commerce fail-open: inactive paid subscriptions no longer inherit live Creator/Pro product truth just because the stored identity plan still says so. Billing state now computes an explicit effective plan, and canceled or past-due accounts resolve to the free contract for entitlements, settings bootstrap, account-tier reporting, and model validation while still keeping legitimately purchased wallet credits.
- That means the backend now separates "historical subscription plan on the identity row" from "current commercial rights". Pro-only image lanes can no longer start for canceled subscriptions, premium chat/export/share entitlements fail closed with the effective free contract, and the wallet still survives for paid credit-pack usage.
- Verification for `.105` also cleaned up a small frontend contract drift that had been breaking `type-check`: `MediaAsset` regained its optional `visibility` field in the web API types, so Studio web is back to passing both `npm run type-check` and `npm run build` on the current billing/bootstrap payload shape.

- `.104` closes a quieter but important commerce/provider drift: free accounts with purchased wallet credits are no longer treated as fallback-only by the image route planner. When managed providers such as Runware are genuinely available, wallet-backed free accounts now promote onto the managed launch lane family instead of being narrated like rescue-lane users by default.
- That same wallet truth now propagates through non-runtime contracts too. Generation credit forecasts, render route previews, and per-model catalog previews now consume the same wallet-backed routing rule, so Studio's backend no longer tells one story in billing/model surfaces and another story at real generation admission time.
- Regression coverage was tightened around this contract. New tests prove wallet-backed free generations prefer Runware when it exists, forecast and experience-contract helpers preserve that same provider truth, and the remaining fallback-only development scenario now has an honest test name instead of implying OpenAI should have been selected.

- `.103` was a focused review-response hardening wave on top of the new commerce doctrine. Paddle webhook verification now unwraps the real secret value, webhook idempotency fingerprints include `event_id`, and missing or unknown billing identities fail closed instead of being silently accepted as successful delivery.
- The same wave closed a few nasty billing integrity gaps: credit-pack checkout kinds can no longer map into paid subscription plans, `subscription.updated` no longer behaves like a free monthly credit refill, and final generation settlement now rejects real credit underflow instead of quietly zeroing balances and granting a free run.
- Service authority also got tighter in small but important ways. Security-log delegation now preserves structured fields, chat rebuild delegations forward their real arguments instead of calling empty stubs, and asset-token signing now refuses to boot outside local development without a configured JWT secret.
- Router safety improved too: demo login is now development-only across the board, `POST /personas` gained a real request model plus mutation rate limiting, and `GET /public/export` now consumes an explicit IP rate limit instead of sitting open-ended.
- `.103` verification held on the touched surfaces: `test_billing_ops` passed in full, targeted Paddle and service-regression shards passed, and the full `test_router_security` file passed again when sharded into two 24-test batches to avoid tool timeout noise.

- Billing backbone truth also moved to `Paddle`.
