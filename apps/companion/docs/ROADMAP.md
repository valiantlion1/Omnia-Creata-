# OmniaCompanion Roadmap

## Strategy

The roadmap should move from a focused text-first companion product to broader media, creator, and ecosystem integrations only after the base experience is proven.

## Phase 0: Product foundation

Goal: lock scope, identity, and data model before implementation expands.

Deliverables:

- canonical product definition
- product boundaries and ecosystem rules
- MVP feature list
- initial implementation handoff

Exit condition:

- the team can start building without reopening basic product identity debates

## Phase 1: MVP

Goal: ship the smallest credible Companion experience.

MVP scope:

- user onboarding
- create a companion
- edit companion profile and personality settings
- text chat with persistent conversation threads
- basic memory and continuity controls
- private companion library
- simple safety and moderation guardrails
- lightweight discovery or featured companion surface

Explicit non-goals for MVP:

- creator marketplace
- deep ecosystem coupling
- native mobile apps
- advanced image generation pipeline
- voice-first interaction
- internal control-plane features

Exit condition:

- a user can create, chat with, revisit, and manage a companion in a way that feels complete enough for repeated use

## Phase 2: Retention and identity depth

Goal: make companions feel more personal and more worth returning to.

Candidate additions:

- richer memory controls
- relationship summaries
- stronger persona editing tools
- better discovery and collection UX
- optional visual identity upgrades
- premium usage limits and subscription shaping

Exit condition:

- retention improves because the experience gets deeper, not just larger

## Phase 3: Creator and ecosystem expansion

Goal: open Companion to more complex creation and publishing flows only after the core product is solid.

Candidate additions:

- creator publishing flows
- shareable companion templates
- marketplace or revenue-share mechanics
- optional Prompt Vault template sync
- optional Studio-powered visual generation
- optional cross-product identity or billing layers

Exit condition:

- the product can support creator and ecosystem complexity without weakening the base companion experience

## Deferred until clearly justified

- OCOS-aware automation flows
- full marketplace governance
- multi-surface desktop/mobile parity
- complex organization or admin constructs
- anything that makes Companion feel like an ops platform

## Open product questions

- should the first shipped version focus on one flagship companion flow or a broader character sandbox?
- should discovery exist in MVP or wait until the private companion loop is proven?
- should image generation start as upload/select only before any generated media support?
- how much memory should be user-controlled versus automatic in the first release?
- should monetization begin with a simple premium tier before any creator economy exists?
