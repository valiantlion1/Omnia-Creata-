# OmniaVault UI/UX Design Tasks

> Last updated: 2026-03-20
> Purpose: Give design tools or designers short, focused tasks derived from the UI/UX development plan
> Use: Send one task at a time, or send several related tasks in parallel

---

## 1. Product Direction

### Task
Design OmniaVault as a real premium mobile application, not a dashboard or marketing website. The result should feel like a product people open daily to capture, organize, and refine ideas without friction.

## 2. Visual System

### Task
Create a complete visual system for OmniaVault with a premium, calm, modern tone. The color palette is open, but it must feel intentional, consistent, mobile-friendly, and suitable for long-term use.

## 3. Typography

### Task
Define a typography system for OmniaVault that feels premium and highly readable. It should clearly separate titles, support copy, content, and metadata without relying on too many font weights or styles.

## 4. Navigation

### Task
Design a simple navigation system for OmniaVault with clear mobile bottom navigation and an app-like desktop shell. The user should always know where they are, how to switch sections, and where the primary action lives.

## 5. Home

### Task
Redesign the Home screen so it guides the user immediately instead of behaving like a dashboard. It should make capture obvious, surface recent work, and help the user continue where they left off.

## 6. Home Empty State

### Task
Design the empty-state version of Home for a brand-new user. It should explain what belongs here and guide the user toward the first capture or first project without feeling like onboarding slides.

## 7. Quick Capture

### Task
Design Quick Capture as the strongest screen in the product. It should feel fast, safe, touch-friendly, and frictionless, with the body field as the main focus and advanced metadata hidden by default.

## 8. Quick Capture Variations

### Task
Create three strong variations of Quick Capture: one ultra minimal, one premium writing-first, and one highly optimized for one-handed mobile use. All three should feel like real production app screens.

## 9. Full Editor

### Task
Design the full editor as a premium writing surface for evolving entries over time. It should prioritize content readability, safe editing, visible save state, and easy access to version history.

## 10. Entry Detail

### Task
Design the entry detail screen as a reading-first surface where content is the hero and metadata is secondary. Actions like edit, duplicate, favorite, archive, and restore history should feel available but not noisy.

## 11. Library

### Task
Redesign the Library as a clean, search-first retrieval screen. It should be easier to scan than the current product, with strong hierarchy, simple filters, and less card overload.

## 12. Search Results

### Task
Design the search results experience so users can quickly tell whether the needed entry exists and why it matches. Show title first, useful content preview second, and quiet metadata third.

## 13. Advanced Filters

### Task
Design an advanced filter surface for Library that keeps the main screen clean. It should support narrowing by type, project, category, favorites, archive state, and recency without overwhelming the user.

## 14. Projects List

### Task
Design the Projects list as a simple, focused project hub rather than a full workspace system. It should show active projects, project summaries, recent activity, and an easy way to create a new project.

## 15. Project Detail

### Task
Design the Project detail screen so a project feels alive but not bloated. It should group related entries clearly, show recent updates, and make it easy to add new work into the project.

## 16. Version History

### Task
Design Version History as a trust-building screen. It should make past changes readable, show when and how versions were created, and make restore feel safe and non-destructive.

## 17. Version Diff

### Task
Design a Version Diff screen that clearly explains what changed between two versions. The comparison should feel readable, premium, and understandable without looking like a developer tool.

## 18. Restore Flow

### Task
Design the restore flow so users understand that restoring an older version does not destroy history. The interaction should feel safe, explicit, and trustworthy.

## 19. Settings

### Task
Redesign Settings as a calm, utility-first screen. It should organize account, theme, language, sync status, release notes, and product preferences without turning into a cluttered feature graveyard.

## 20. Profile And Account

### Task
Design the profile/account surface so it feels like part of the product, not a generic settings form. It should communicate identity, plan state, and account control clearly.

## 21. Sync Status

### Task
Design a sync status screen or panel that explains current sync state in plain language. It should reassure the user when things are healthy and clearly explain pending or failed sync states.

## 22. Conflict Resolver

### Task
Design the conflict resolution flow for offline or multi-device changes. It should help users understand what changed, preserve both versions when needed, and avoid any feeling of silent data loss.

## 23. Release Notes

### Task
Design an in-app release notes screen that feels simple and polished. It should present current and previous updates cleanly, without looking like a changelog dump.

## 24. Onboarding

### Task
Design a concise onboarding flow that explains the product simply and quickly. It should teach capture, revisit, projects, and version safety without sounding like marketing copy.

## 25. Splash

### Task
Design a premium splash screen for OmniaVault. It should be short, elegant, and app-like, with motion that feels intentional but never theatrical.

## 26. Sign Up

### Task
Design the sign-up screen as part of the product family, not a generic auth page. It should feel clean, safe, and easy to use on mobile.

## 27. Sign In

### Task
Design the sign-in screen to feel premium and trustworthy. It should be visually simple, easy to use, and consistent with the rest of the app system.

## 28. Forgot Password

### Task
Design the forgot-password flow so it feels calm and understandable. The user should immediately know what to do and what happens next.

## 29. Offers / Upgrade

### Task
Design an upgrade or offers surface that explains product value without aggressive sales behavior. It should feel premium, believable, and secondary to the core product experience.

## 30. AI Assist

### Task
Design the AI assist layer as a secondary helper system, not a chat app. It should support summarize, improve, classify, and suggest actions in a way that feels integrated but never dominant.

## 31. AI Apply / Reject

### Task
Design the AI suggestion review flow so users can preview, apply, or reject changes with confidence. The interaction should reinforce that the user remains in control.

## 32. Tutorial Tips

### Task
Design lightweight tutorial surfaces for Home and Editor that teach the product without getting in the way. They should feel useful, temporary, and easy to dismiss.

## 33. Export / Share

### Task
Design an export/share surface that feels like a clean system utility. It should communicate what can be exported or shared without turning into a marketplace or publishing flow.

## 34. Archive / Delete Confirmation

### Task
Design archive and delete confirmation patterns that clearly distinguish reversible from destructive actions. Archive should feel light; delete should feel more deliberate.

## 35. Save Feedback

### Task
Design save-state feedback patterns for draft, saving, saved, sync pending, and sync failed. Users should always know whether their work is safe.

## 36. Empty States

### Task
Design empty states for Home, Library, Projects, Version History, and Search. Each empty state should explain what belongs there and point the user toward the next useful action.

## 37. Loading States

### Task
Design loading states and skeletons that match the actual layouts of the product. They should feel intentional and premium, not like generic placeholders.

## 38. Error States

### Task
Design error states for auth, sync, search, and network issues. They should be calm, clear, and helpful without sounding technical or alarming unless the situation truly warrants it.

## 39. Motion System

### Task
Define a motion system for OmniaVault that includes splash, route transitions, button press states, bottom sheets, save feedback, and selected states. Motion should be short, subtle, and support clarity.

## 40. Micro-Interactions

### Task
Design micro-interactions for save, favorite, pin, archive, restore, and chip selection. These interactions should make the product feel alive without becoming distracting.

## 41. Component System

### Task
Design a complete component system for OmniaVault including buttons, chips, inputs, textareas, cards, list rows, nav items, sheets, dialogs, toasts, and version rows. Every component should define default, focus, pressed, selected, and disabled behavior where relevant.

## 42. Mobile Behavior

### Task
Design the app for mobile-first use with thumb-friendly primary actions and reliable keyboard behavior. Every important screen should feel excellent on a phone before desktop adjustments are added.

## 43. Tablet Behavior

### Task
Design tablet adaptations that add breathing room and structure without changing the identity of the app. Tablet should feel like a refined extension of mobile, not a different product.

## 44. Desktop Behavior

### Task
Design desktop layouts that still feel like an application rather than a website. Avoid giant hero sections, giant empty spaces, or dashboard-like admin layouts.

## 45. Accessibility

### Task
Design OmniaVault so that accessibility is part of premium quality. Ensure readable contrast, clear focus states, strong touch targets, understandable labels, and reduce-motion support.

## 46. PWA And Android Shell

### Task
Design the app shell with installed-app behavior in mind. It should feel correct in a browser, an installed PWA, and an Android wrapper, including safe-area spacing and splash behavior.

## 47. Scope Guard

### Task
Redesign the product while explicitly avoiding seller dashboards, creator payouts, marketplace views, prompt-selling flows, and other creator-economy patterns. Keep the product focused on capture, organization, revisit, and version-safe editing.

## 48. Final Cohesion Pass

### Task
Perform a full cohesion pass across all designed screens so they feel like one product family. Unify spacing, typography, hierarchy, icon behavior, interaction patterns, empty states, and motion language.
